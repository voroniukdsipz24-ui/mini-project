# Vision Document — Hotel Booking System

## 1. Назва проєкту

**HotelBooking** — Система бронювання готелю «Grand Palais»

---

## 2. Проблема

Невеликі готелі часто ведуть бронювання вручну (Excel, папір), що призводить до:
- подвійних бронювань одного номера на ті ж дати;
- помилок у розрахунку вартості (ігноруються сезон, тип номера);
- відсутності оперативної аналітики: менеджер не знає поточну завантаженість.

---

## 3. Цільові користувачі

| Роль | Опис |
|------|------|
| **Адміністратор готелю** | Головний користувач. Реєструє гостей, створює та управляє бронюваннями через веб-інтерфейс. |
| **Менеджер** | Переглядає звіти: завантаженість, дохід, ТОП гостей. |

---

## 4. Основні сценарії використання

### UC-1: Пошук вільних номерів і бронювання
**Актор**: Адміністратор
**Передумова**: гість вже зареєстрований у системі
**Основний шлях**:
1. Адміністратор відкриває розділ «Номери», бачить сітку всіх номерів з фільтрами
2. Натискає «+ Нове бронювання», обирає гостя з dropdown
3. Обирає номер з dropdown (показуються лише доступні)
4. Вводить дати заїзду/виїзду — система live-розраховує ціну (базова ціна × ночей × сезон)
5. Натискає «Створити бронювання»
6. Система перевіряє відсутність конфліктів, зберігає, номер → Reserved
7. Toast-нотифікація про успіх, таблиця оновлюється

**Альтернативний шлях**: якщо номер недоступний — toast (червоний) з повідомленням.

---

### UC-2: Повний lifecycle бронювання
**Актор**: Адміністратор
**Передумова**: бронювання у статусі Pending/Confirmed/CheckedIn
**Основний шлях**:
1. У таблиці бронювань адміністратор бачить кнопки дій залежно від статусу
2. Pending → клік «Підтвердити» → статус Confirmed
3. Confirmed → клік «Check-in» → CheckedIn, номер → Occupied
4. CheckedIn → клік «Check-out» → CheckedOut, номер → Available, оплата → Paid

**Альтернативний шлях**: спроба недопустимої дії → 400 Bad Request з зрозумілим повідомленням, toast (червоний).

---

### UC-3: Перегляд аналітичних звітів
**Актор**: Менеджер
**Основний шлях**:
1. Перехід в розділ «Звіти»
2. Бачить три картки: дохід за типами номерів (бари), ТОП гостей (бари), зайнятість за місяцями (стовпчастий графік)
3. Дані оновлюються автоматично з API

---

### UC-4: Реєстрація нового гостя
**Актор**: Адміністратор
**Основний шлях**:
1. Натискає «+ Новий гість» (на сайдбарі або в розділі «Гості»)
2. Заповнює форму: ім'я, прізвище, email, телефон, паспорт, дата народження
3. Система перевіряє унікальність email
4. Гість зберігається, з'являється у списку

---

### UC-5: Скасування бронювання
**Актор**: Адміністратор
**Основний шлях**:
1. У таблиці бронювань натискає «Скас.» (доступно для Pending/Confirmed)
2. Введення причини у prompt
3. Бронювання → Cancelled, номер → Available
4. Якщо бронювання у CheckedIn — кнопка не показується; спроба DELETE через API → 400

## 5. Нефункціональні вимоги

| №  | Вимога | Деталі |
|----|--------|--------|
| NFR-1 | **Тестованість** | Бізнес-логіка ізольована від I/O. Application-сервіси залежать лише від інтерфейсів, тому тестуються з in-memory fake-репозиторіями без файлів/БД та без HTTP. |
| NFR-2 | **Відновлення після помилки** | Жоден некоректний ввід не призводить до краша. Доменні винятки перетворюються в HTTP 400 з зрозумілим JSON-повідомленням; GUI відображає toast. |
| NFR-3 | **Простота розширення** | Додавання нового use case = метод у Application service + endpoint у Web/Program.cs. Domain і Infrastructure не торкаються. |
| NFR-4 | **Розділення UI і бекенду** | Frontend — статичний HTML/CSS/JS, спілкується через REST. Можна замінити на React/Vue/мобільний додаток без зміни бекенду. |
| NFR-5 | **Зберігання стану** | JSON-файли у `data/`. Атомарний запис (tmp-файл + rename) — захист від часткових даних при збої. |

---

## 6. Архітектурне рішення: Web + Console

Проєкт містить **два варіанти presentation-шару**, обидва на одних і тих самих Application-сервісах:

| Варіант | Призначення |
|---------|-------------|
| **HotelBooking.Web** | Основний інтерфейс: браузер + REST API |
| **HotelBooking.Console** | Lab 34 baseline: демонструє чистоту архітектури — той самий бекенд через інший UI |

Це доводить: Domain і Application повністю незалежні від UI. Можна додати мобільний клієнт або Blazor SPA без жодних змін у бізнес-логіці.

---

## 7. Обмеження

- Single-tenant (один готель)
- Без платіжної інтеграції
- Без email-нотифікацій
- Без авторизації (single-admin context для MVP)
